# YNX Grant Application Kit

Status: active working template  
Prepared on: 2026-06-27  
Use: adapt for ecosystem grants, public-goods funding, or builder support applications

## 1. Short application summary

```text
YNX is an early open-source infrastructure project building AI-native execution,
policy-bounded Web4 tooling, machine-settlement rails, and accountability /
forensics infrastructure. The strongest proof today is public technical
evidence: live public-testnet services, public repository history, public
documentation, and conservative disclosure around what is live versus what is
still incomplete.

This application is not asking reviewers to fund a paper architecture. It is
asking for support to harden and package an already-public technical stack into
more useful infrastructure for builders, operators, and ecosystem participants.
```

## 2. Public-good framing

```text
YNX is best evaluated today as infrastructure and public-goods-oriented tooling.
The most relevant outputs are open and reusable: policy-bounded execution
patterns, operator and traceability tooling, accountability / forensics
surfaces, and machine-payment / settlement primitives that can help other
builders work more safely.
```

## 3. Problem statement

```text
Early onchain and agentic systems are often hard to inspect, hard to constrain,
and described more aggressively than their true state of readiness. Builders
need infrastructure that is easier to verify, safer to operate, and more
disciplined about execution boundaries. YNX is being built to address that gap
through policy controls, session authorization, AI settlement rails, and
accountability tooling.
```

## 4. What exists today

Use this checklist selectively depending on the form:

- live public-testnet chain infrastructure
- live RPC / indexer / explorer / AI gateway / Web4 hub / website surfaces
- AI/Web4 policy/session boundaries
- machine-payment vault / job / settlement rails
- protected trace and forensics case flow
- lot-lineage tracing and taint-model support
- provenance anchors (`issuance_id`, `deposit_batch_id`)
- public-versus-protected evidence boundaries

## 5. Honest limitations

Use directly when needed:

```text
YNX should not currently be described as a completed production financial
network, a fully formed operating company, or a fully completed multi-route
bridge platform. The strongest current live bridge truth remains 2/5
full-loop-tested routes. This application is therefore best understood as a
request to harden and extend real public infrastructure, not to paper over open
gaps.
```

## 6. Use of funds template

```text
Requested support would be used for:

1. Security and funds-safety hardening
2. Observability, persistence, and operational resilience
3. Builder tooling, docs, and ecosystem-specific integration work
4. Accountability / forensics packaging and operator workflows
5. Company-readiness basics such as legal, privacy, and policy implementation
```

## 7. Milestone template

```text
Milestone 1 — Hardening and packaging
- tighten security boundaries on AI/Web4/trace surfaces
- complete reproducible status and evidence reporting
- publish operator and builder documentation updates

Milestone 2 — Ecosystem-facing integration work
- build or adapt ecosystem-specific tooling/integration layer
- document why the work is useful to this ecosystem
- publish public implementation notes or examples

Milestone 3 — Accountability and evidence layer
- improve trace/accountability outputs
- strengthen provenance/evidence packaging
- publish reusable operator / builder guidance
```

## 10. Do not say

Avoid:

- `all bridge routes are complete`
- `production-ready financial network`
- `fully audited`
- `fully legally complete`
- `institution-ready`
- any wording that implies custody or licensed redemption where none exists
