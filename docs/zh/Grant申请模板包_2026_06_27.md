# YNX Grant 申请模板包

状态：active working template  
整理日期：2026-06-27  
用途：适配 grant、public-goods funding、builder support 申请

## 1. 短版申请摘要

```text
YNX 是一个早期但已经有公开技术证据的基础设施项目，方向是 AI-native
execution、policy-bounded Web4 tooling、machine settlement rails，以及
accountability / forensics infrastructure。今天最强的证明不是概念图，而是
公开代码、公开文档、live public-testnet 服务，以及对“哪些是 live、
哪些还没完成”的保守披露。

这次申请不是在为一套纸面架构融资，而是在请求支持一个已经公开存在的技术栈，
让它在 builder、operator 和 ecosystem participant 视角下变得更硬、更好用。
```

## 2. Public-good 表述

```text
YNX 当前最适合被评估为基础设施和 public-goods 风格 tooling。最有价值的
输出不是单纯服务 YNX 自己，而是可复用的执行边界模式、operator /
traceability tooling、accountability / forensics surface，以及更安全的
machine-payment / settlement primitive。
```

## 3. 问题陈述

```text
很多早期 onchain 和 agentic 系统都很难被检查、很难被约束，而且对外表述
通常比真实成熟度更激进。Builder 需要的是更容易验证、更安全、更纪律化的
执行基础设施。YNX 正在通过 policy control、session authorization、AI
settlement rail 和 accountability tooling 去解决这个问题。
```

## 4. 今天已经存在的东西

按表格需要挑选使用：

- live public-testnet chain infrastructure
- live RPC / indexer / explorer / AI gateway / Web4 hub / website
- AI / Web4 policy/session boundary
- machine-payment vault / job / settlement rail
- protected trace / forensics case flow
- lot-lineage tracing + taint model
- provenance anchors（`issuance_id`、`deposit_batch_id`）
- public / protected evidence boundary

## 5. 诚实边界

可以直接这样写：

```text
YNX 现在不应该被描述成已经完成的 production financial network、已经设立
完毕的 operating company，或者已经完成的 multi-route bridge platform。
当前最强的 live bridge truth 仍然只是 2/5 full-loop-tested routes。
因此这次申请最适合被理解为：给一个已经公开运行的技术栈做 hardening 和
extension，而不是掩盖仍然存在的开放问题。
```

## 6. Use of funds 模板

```text
资金将优先用于：

1. Security 与 funds-safety hardening
2. Observability、persistence 与 operational resilience
3. Builder tooling、docs 和 ecosystem-specific integration
4. Accountability / forensics packaging 与 operator workflow
5. Legal、privacy、policy 等 company-readiness 基础动作
```

## 7. Milestone 模板

```text
Milestone 1 — Hardening and packaging
- 加固 AI / Web4 / trace 安全边界
- 完成可重复状态与证据输出
- 发布 operator / builder 文档更新

Milestone 2 — Ecosystem-facing integration
- 做生态特定的 tooling / integration layer
- 文档化它为什么对该生态有用
- 发布公开实现说明或示例

Milestone 3 — Accountability and evidence layer
- 继续增强 trace / accountability 输出
- 加强 provenance / evidence packaging
- 发布可复用 operator / builder 指南
```

## 8. 各生态申请角度提示

### 8.1 Ethereum Foundation ESP

```text
这份申请最强的角度是 public-goods infrastructure：更安全的 onchain
execution 与 accountability tooling。它不是单纯的 YNX feature set，
而是一套围绕 policy-bounded machine actions、traceability、operator-facing
evidence 的可复用实现方向。
```

### 8.2 Base

```text
这份申请更适合从 developer-facing infrastructure 角度切入：
agent payments、execution boundary、operator / accountability tooling。
重点不是“我们也是一条链”，而是“这对 Base builder 有什么具体价值”。
```

### 8.3 Optimism / Superchain

```text
这份申请最适合被描述成 open tooling / public-goods infrastructure，
而且要强调它不只是对 YNX 自己有用，而是对更广的 onchain operator /
builder 世界也有复用价值。
```

## 10. 不要写的话

避免：

- `all bridge routes are complete`
- `production-ready financial network`
- `fully audited`
- `fully legally complete`
- `institution-ready`
- 任何暗示 custody 或 licensed redemption 已经到位的话术
