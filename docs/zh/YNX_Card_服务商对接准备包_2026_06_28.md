## 1. YNX 希望服务商先理解什么

YNX 现在不是在拿一个已经完成的真实发卡项目去要求马上上线。

YNX 现在提供的是一套已经证明了下列能力的可编程消费控制底座：

- owner 范围控制
- policy-bounded agent 权限
- 类卡片消费规则判断
- 明确的 approve / decline 结果
- 可审计的事件历史

所以这份包的目的不是“拿一个 mock 要求明天发真卡”，而是：

- 让服务商先 review 当前控制面逻辑
- 对齐真实接入还缺哪些条件
- 把 YNX 控制层映射到服务商的 issuer / processor 模型
- 提前识别法务、合规、PCI、清结算、运营要求

- freeze card
- resume card

## 3. 现在还没有什么

YNX 当前**不声称**已经有：

- 真实发出的卡
- live PAN / CVV 处理
- PCI 敏感卡数据生产环境
- 已经打到 Visa / Mastercard / issuer processor rail 的 live 授权
- provider 级清结算 / 对账 / dispute / refund 流程
- 由 YNX 基础协议公司默认承担的消费者 KYC 主营业务

这份准备包是刻意的 pre-provider、pre-production。

## 4. YNX 想要的对接模型

YNX 希望保留下面这层清晰分工：

- YNX control plane 负责：
  - wallet / policy / session / agent logic
  - programmable spend constraints
  - audit trail
  - operator review / accountability surface
- provider / issuer 侧负责：
  - 发卡
  - 受监管 card program 运营
  - 如有需要的 KYC / KYB / sanctions 检查
  - network connectivity
  - settlement / reconciliation / dispute 运营
  - PCI 范围卡数据处理

也就是说，YNX 应该是可编程授权和审计层，而不是默认变成隐藏托管方或 issuer operator。

## 5. 能力映射表

| YNX 需求 | 当前 YNX 状态 | 未来真实 provider 侧职责 |
|---|---|---|
| 可编程消费规则 | mock 已实现 | 提供能够执行或配合外部规则引擎的授权控制 |
| agent-bounded authorization | mock 已实现 | 支持映射 cardholder / business / delegate 模型，同时不破坏 issuer 控制 |
| merchant / MCC / country filter | mock 已实现 | 映射到 processor/issuer 控制或 pre-auth policy engine |
| 单笔 / 日 / 总限额 | mock 已实现 | 通过 issuer 限额、shadow ledger 或 provider 控制实现 |
| audit trail | mock 已实现 | 提供 reference id、event reason、reconciliation anchor |
| freeze / resume | mock 已实现 | 提供 provider 侧 suspend/resume 能力 |
| 发卡 | 未实现 | provider 负责 |
| PAN/CVV 生命周期 | 未实现 | provider 在 PCI 范围内负责 |
| KYC/KYB/AML | YNX 基础协议默认不做 | 受监管合作方负责，除非法务结构改变 |
| settlement / disputes / refunds | mock ledger 已实现，provider 级运营未实现 | provider 负责，并与 YNX / operator 流程对接 |

## 6. 真实接入前还需要的技术条件

从 mock 进入真实 provider 之前，YNX 还需要：

- provider API contract review
- provider API 的 auth 模型
- provider 侧幂等和 reference mapping
- webhook 或 polling 的事件模型
- provider credentials 的安全存储模型
- provider 级 reconciliation ledger 和 exception handling
- refund / reversal / dispute / incident workflow
- 把 YNX `card.authorized` / `card.declined` / `card.settled` / `card.refunded` 事件和 provider event id 对齐的审计映射

## 7. YNX 想保留的安全边界

YNX 希望保留这些不变项：

- public demo 里不出现生产私钥或 provider credentials
- provider 真正上线前，不用误导性的 “live card” 语言
- 当前 mock 层不存 PAN / CVV / cardholder data
- 不把协议公司默认扩张成消费者 KYC 业务
- 明确区分：
  - testnet
  - mock
  - sandbox provider integration
  - production provider integration

## 8. 合规和法务前置条件

任何真实 provider go-live 前，YNX 还需要：

- 法律实体和签约主体
- 律师审过的经营边界
- KYC / KYB / sanctions / AML 职责分配
- 数据处理和隐私 review
- PCI scope 决策和控制权归属
- incident response 和 customer support 责任模型
- settlement / treasury / reserve / reconciliation 责任模型

## 9. 服务商下一步可以回答的问题

1. 服务商能否支持 YNX 想要的这种细粒度 programmable authorization control？
2. merchant / MCC / country / amount 约束，provider 能否直接支持，还是要 YNX 侧做 shadow pre-authorization layer？
3. YNX 应该持久化哪些 provider 标识，才能把 provider authorization 和 YNX audit event 对齐？
4. sandbox access 之前需要哪些 KYB / 合同 / 合规前置？
5. provider 能否支持企业 / delegate / AI-agent bounded spending 模型？
6. 按拟议架构，YNX 最终会继承多大的 PCI / cardholder-data 范围？

## 10. 最合适的下一步

下一步不是直接 public launch。

最合适的下一步是拿这份准备包去做一轮 scoped provider conversation，并同时附上：

- 当前 capability audit
- 当前 truth matrix
- 当前 card mock demo evidence（`reports/ynx_card_demo_latest/`）
- 当前 compliance boundary docs

这才是从“mock logic 已证明”走向“真实 provider integration 负责任规划”的正确桥梁。
