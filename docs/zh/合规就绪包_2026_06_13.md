# YNX 合规就绪包

状态：working compliance packet  
最后更新：2026-06-14  
规范语言：中文

## 用途

这份文档是给创始人、律师、投资人和 grant reviewer 的实用合规包。
它不是法律意见，而是把 YNX 当前最合理的经营边界、公司结构选择和待补动作
整理成一个可以直接尽调和落地的版本。

## 1. 当前建议的经营定位

建议继续坚持以下边界：

- 非托管基础设施公司；
- 不直接保管用户资产；
- 不宣称自己是中心化交易所或撮合引擎；
- 不宣称自己发行稳定币或管理稳定币储备；
- 不把消费者 KYC 做成默认产品线；
- 不对 NYXT 做投资收益承诺。

这和 `docs/zh/YNX_非托管商业与合规边界.md` 已有口径一致，应继续作为
融资和商业落地的默认边界。

## 2. 司法辖区和主体建议

默认建议结构：

- 新加坡运营公司作为主要签约和基础设施主体；
- 如以后需要将治理、公共物品或 grants 与运营实体分开，再考虑独立
  foundation / governance entity；
- 只有当美国融资、美国企业销售或美国团队扩张成为一线需求时，再考虑
  Delaware C-Corp。

这样做的原因：

- 当前基础设施和叙事已经天然偏向新加坡；
- 项目更容易被定义为 infrastructure software，而不是受监管金融服务；
- 但任何涉及 digital-token services 的扩展，都必须重新让律师审查。

## 3. 需要律师核对的官方材料

这些官方来源应进入法务核查包：

- MAS 于 2025-06-06 发布的 DTSP 澄清：
  [MAS Clarifies Regulatory Regime for Digital Token Service Providers](https://www.mas.gov.sg/news/media-releases/2025/mas-clarifies-regulatory-regime-for-digital-token-service-providers)
- MAS 关于 DTSP 的 AML/CFT notice：
  [FSM-N27 AML/CFT - DTSPs](https://www.mas.gov.sg/regulation/notices/fsm-n27-amlcft---dtsps)
- ACRA 本地公司注册指引，2026-05-16 更新：
  [Registering a local company via Bizfile](https://www.acra.gov.sg/register/business/registering-different-business-structures/local-company/registering-via-bizfile/)
- PDPC 的 DPO 上手指南，2026-04-14 发布：
  [Kickstart Your Data Protection Journey](https://www.pdpc.gov.sg/organisations/resources/getting-started-as-a-data-protection-officer-dpo)
- PDPC 的 PDPA 概览：
  [PDPA Overview](https://www.pdpc.gov.sg/about/the-legislation/pdpa-overview)

解释口径：

- MAS 在 2025-06-06 的澄清意味着，只要一个主体被认定为 digital token
  service provider，并且从新加坡开展相关业务，即便服务对象主要在海外，
  也不能随意假设“无需牌照”。
- 所以 YNX 在没有律师确认前，不应轻易把自己扩张叙述为 custody、
  exchange、dealing 或其他 token service 业务。

## 4. 近期必须完成的公司动作

至少应完成：

1. 设立运营主体并明确责任人。
2. 完成名称预留和 ACRA 注册流程。
3. 建立会计、税务、记账和合同归属。
4. 指定 DPO 负责人并形成内部流程。
5. 发布 privacy policy、terms of use、risk disclosures、security contact。
6. 建立 incident response 和 vulnerability disclosure 流程。
7. 如需要对接企业客户，建立 sanctions / KYB 流程。
8. 在律师确认前，不把消费者 KYC 纳入协议主营范围。

这些事项在当前仓库里仍应被视为待完成动作，而不是已经完成的事实。

## 5. 产品表述的允许项与禁用项

允许：

- live public testnet；
- 托管基础设施；
- AI settlement testing infrastructure；
- policy-bounded agent execution；
- EVM-compatible builder surfaces；
- 测试网 bridge 证据；
- 在文档和 route 证据都存在前提下的有限公开测试网 wrapped-asset / swap 测试。

禁用，除非律师和生产控制都到位：

- 托管；
- 交易所；
- broker / dealer 口径；
- 可赎回稳定币口径；
- 生产级外部资产赎回承诺；
- “全球都合规”这类泛化说法；
- 代币收益或投资回报承诺。

## 6. 数据保护基线

只要 YNX 接触到下面这些信息，就应该按 PDPA 风格管理：

- 账户邮箱；
- 与可识别用户相关的日志；
- 企业客户联系人；
- 工单和支持记录；
- 与钱包或账户关联的分析数据。

最低动作：

- 指定 DPO owner；
- 定义 retention / deletion policy；
- 限制日志保留和访问权限；
- 盘点第三方云和数据处理商；
- 发布外部隐私声明；
- 定义 breach escalation 和通知流程。

## 7. 代币与金库披露

融资和对外材料中应统一说明：

- NYXT 是 utility / governance asset，不是投资收益承诺；
- YUSD.test 是不可赎回的 synthetic test asset；
- 公测网 wrapped assets 是测试表示，不是受监管生产托管或赎回承诺；
- 公司不能把 treasury 账面价值当作生存基础。

## 8. 融资或大规模对外前必须具备的材料

- 公司 / 项目总览；
- 架构与 live status 总览；
- risk disclosure 页面；
- terms of use 草案；
- privacy policy 草案；
- non-custodial boundary 说明；
- security disclosure process；
- 融资备忘录；
- diligence index；
- 合规就绪包。

## 9. 目前仍然存在的红旗

- 还没有外部律师正式 legal memo；
- 还没有生产级审计意见；
- 截至 2026-06-14，`https://www.ynxweb4.com/privacy` 与 `/terms` 仍返回网站壳，而不是独立 legal disclosure 页面；
- `SECURITY.md` 目前以 GitHub Advisory 为正式入口，专用公司级 security alias 仍是待完成动作；
- bridge 相关叙事必须继续严格限定在 testnet 范围；
- BSC / BNB 路线不应该再主导融资和合规定位。

## 10. 标准免责声明

建议统一使用：

`YNX 当前是公共测试网基础设施项目。在获得牌照、审计和律师明确批准前，
应将其描述为 non-custodial software and hosted infrastructure，而不是
受监管金融服务。`
