# YNX 安全响应策略

状态：active working policy  
最后更新：2026-06-14  
规范语言：中文

## 目的

这份文件把 YNX 的安全处理方式，从“开源项目联系人”升级成“可重复执行的
最小运行策略”。它不会假装 YNX 已经有 24/7 机构级安全团队；它的作用是让
项目在从公开测试网走向商业化基础设施的过程中，至少具备可信、诚实、可追
踪的安全响应流程。

## 1. 当前成熟度说明

今天真实成立的事：

- YNX 有可用的私下漏洞披露入口；
- YNX 有可被实际演练和回归测试的公网基础设施；
- YNX 能修代码、改 runbook、发布 operator 侧修复。

今天还不成立的事：

- 还没有正式 bug bounty；
- 还没有覆盖全栈的第三方最终审计意见；
- 还不能承诺 24/7 值班响应；
- 不能把公开测试网 bridge route 说成生产级托管安全能力。

所以这是一份“测试网阶段的正式安全流程”，不是“已经机构级安全成熟”的证
明。

## 2. 漏洞入口

首选私下入口：

- GitHub Security Advisory

备用入口：

- 通过官网发起私下联系，并明确请求安全披露通道

当前成熟度补充说明：

- GitHub Advisory 是目前唯一已明确文档化的专用入口；
- 专用公司级 security alias 仍是待完成控制项，不应被表述成已经建成。

建议报告内容：

- 受影响组件；
- 影响说明；
- 复现步骤或 PoC；
- 是否已经公开或疑似被利用；
- 方便跟进的联系方式。

## 3. 严重级别

### Critical

示例：

- 未授权 mint、release、settlement、policy bypass；
- 公网基础设施远程执行；
- bridge 或 signer 被滥用，导致未授权资产流动；
- AI/Web4 受保护动作被绕过。

处理目标：

- 看到后尽快确认；
- 维护者在线时当天完成分级；
- 在广泛披露前尽力先完成缓解。

### High

示例：

- 持久化提权；
- 与公网服务相关的可利用数据暴露；
- 可造成实质影响的 replay、伪造结果、session 滥用。

处理目标：

- 3 个工作日内确认；
- 可复现时 7 个工作日内给出修复计划。

### Medium / Low

示例：

- DoS 加固缺口；
- 非默认配置风险；
- 会把 operator 引向不安全部署方式的文档问题。

处理目标：

- 进入正常修复节奏；
- 在可行时补测试或补文档。

## 4. 响应流程

1. 私下确认收到。
2. 复现并分级。
3. 判断是否需要先做 operator 侧缓解。
4. 修补代码，并在可行时加回归测试。
5. 如果操作方式变了，同步更新文档和 runbook。
6. 在安全时机发布 advisory 或变更说明。

## 5. 公开测试网特别规则

- 测试网资产绝不能表述成生产资金。
- bridge route 证据必须明确限定为 testnet。
- 如果漏洞影响安全表述，应该立即降级对外说法，而不是继续模糊表述。
- 如果服务退化，应该优先诚实地更新状态和 operator 说明，而不是默默部分失效。

## 6. 最低操作证据

YNX 应保持以下内容可用且最新：

- `SECURITY.md`
- 已知鉴权与传输层加固问题的回归测试
- public readiness 与 security-gate 探针报告
- 需要 operator 改动作时的影响说明

## 7. 仍需补齐的缺口

在宣称“机构级安全就绪”前，YNX 仍需要：

- 覆盖链关键路径和 bridge 关键路径的外部审计；
- 要求硬件签名或等价控制的密钥管理策略；
- 不依赖单个创始人的值班与升级责任；
- 有记录的 incident / rollback drill；
- 组织级而不是个人级的安全联系地址。

## 8. 对外固定句式

需要统一口径时，使用这句：

`YNX has a functioning private disclosure path and active regression hardening,
but it is still a public-testnet infrastructure stack and should not be
described as fully institution-grade security operations yet.`
