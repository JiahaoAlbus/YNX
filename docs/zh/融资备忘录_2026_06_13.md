# YNX 融资备忘录

状态：investor diligence draft  
最后更新：2026-06-14  
规范语言：中文

## 1. 一句话总结

YNX 是一个已经上线的 Web4 公共测试网执行层，面向人和 AI agent，
核心是策略约束执行、EVM 兼容入口，以及可验证的 AI 任务结算基础设施。

## 2. 今天已经有什么

截至 2026-06-13，YNX 已经具备：

- 已上线的 RPC、EVM RPC、REST、faucet、indexer、explorer、AI gateway、
  Web4 hub；
- 4 个 bonded validator 的公共测试网；
- owner-policy-session 的受保护 agent 执行流程；
- vault / job / result / finalize 的 AI settlement rail；
- `5/5` 已 full-loop-tested 的测试网 bridge route，其中只有 `2/5` 拥有最强的公开自动化实证；
- 一套可运行的 operator 脚本、探针、runbook 和 readiness 证据。

所以 YNX 已经不属于“只有概念图和 PPT”的阶段。现在最核心的事情不是
证明技术上能不能做，而是把线上能力收束成一个可融资、可商业化、可合规
表述的公司。

但截至 2026-06-14，仍有几项尽调可见阻塞没有闭环：

- 还没有外部安全审计报告；
- 还没有外部律师 legal memo；
- 官网 `/privacy` 和 `/terms` 还不是独立可访问的法律披露页；
- 公司主体、明确责任人以及专用 security/contact alias 仍属于待完成动作。

## 3. 投资逻辑

YNX 站在四个长期需求的交叉点上：

1. AI agent 需要受策略约束的执行，不只是能签钱包。
2. 链上应用需要低摩擦的 EVM 接入和强 operator 控制。
3. 机器支付和任务结算需要可审计的生命周期基础设施。
4. 基础设施公司必须有不依赖代币投机的收入模型。

最适合 YNX 的讲法是：非托管基础设施公司 + 协议放大能力，
而不是“代币会涨”的融资故事。

这份备忘录不应暗示：

- 现在已经有 novel consensus moat；
- 现在已经是 mainnet-ready 的金融基础设施；
- 现在已经具备 production bridge / production custody 能力。

## 4. 为什么是现在

市场正从“AI 能建议”进入“AI 能执行”的阶段。这个阶段对下面这些能力的
需求会越来越强：

- 执行前的 policy enforcement；
- 可审计的 session scope；
- machine budget control；
- task / result / payment settlement rail；
- 更安全的 AI-assisted operator workflow。

YNX 这些基础原语已经在公共测试网上线，因此融资逻辑可以建立在“线上证据”
而不是纯故事上。

但最强的线上证据发生在 execution / policy 层，不在通用 L1 组装层。

## 4.5 为什么仍然保留链形态

最诚实的答案不是“因为共识本身就是护城河”。

更诚实的答案是：

- YNX 希望把 policy、session scope、execution 和 settlement evidence
  放在同一个可验证执行面上；
- YNX 受益于统一控制 developer/operator stack；
- 如果市场最后只认可 middleware 价值，YNX 也应先把自己定义成执行基础设
  施公司，而不是硬讲成底层共识故事。

这意味着链层叙事还需要通过真实采用去赢得，而不是天然享受估值溢价。

## 5. 本轮资金真正该投什么

应优先投入的核心范围：

- Web4 policy 和 session 基础设施；
- AI settlement 和 machine payment rails；
- 托管 RPC / indexer / explorer / gateway；
- builder 工具、SDK 和 operator observability；
- 仅在确实服务开发者场景时，有限推进 wrapped asset / bridge 支持，并保持所有对外表述严格限定在 testnet 范围。

明确降级的范围：

- 任何依赖 BNB Chain developer support 才成立的融资叙事；
- 任何把 BSC route completion 当作公司可投前提的说法；
- 任何暗示已经具备生产级外部资产托管或赎回能力的营销。

BSC / BNB 相关路线可以保留为可选技术 backlog，但不应该再成为融资故事、
公司设立逻辑或近期 use of funds 的必要前提。

## 6. 资金用途

本轮资金最应该覆盖：

- 安全审计、修复和 incident-response hardening；
- 生产级 observability、告警、备份和灾备；
- 开发者体验、文档和 SDK 成熟度；
- validator 多样性和外部 operator onboarding；
- 法务主体、条款、隐私、合规落地；
- hosted infra 和 enterprise Web4 / AI 场景的试点客户。

## 7. 收入模型

YNX 的商业模式应该以非投机收入为主：

- 托管 API 和基础设施订阅；
- 专属或私有部署；
- SLA 与企业支持合同；
- validator、监控和灾备工具；
- AI 执行基础设施与 machine-payment 工作流；
- 生态项目的审计、集成和认证服务。

NYXT 应该是 utility 和协调层，而不是公司生存的唯一前提。

## 8. 投资人需要明确知道的事

- YNX 现在是 live public testnet，还不是 mainnet-grade production network。
- YNX 今天最强的护城河是 Web4 policy/session 和 AI settlement 栈，不是通
  用 Cosmos/EVM 组装。
- EVM 兼容解决的是接入摩擦，不是核心差异化。
- validator 去中心化在改善，但还不够支撑激进的 decentralization claim。
- 公司默认应保持 non-custodial 边界。
- wrapped testnet routes 是架构证据，不是受监管生产托管的证明。
- `automatic_loop_ready` 只是测试网配置与 adapter 就绪状态，不等于生产安全或规模化已证实。
- 仓库里虽然有历史 tokenomics 草案，但那不是当前融资 term sheet，也不是必然的 founder-fee 承诺。
- 最好的融资逻辑是基础设施硬化和商业化，而不是流动性表演。

## 9. 融资口径

YNX 适合对接：

- pre-seed / seed 基础设施投资人；
- 支持 open protocol tooling、safe agent execution、public infra 的 grant；
- 关注 AI action safety、machine payment、enterprise Web3/Web4 rail 的战略伙伴。

建议统一表达：

`We are raising to harden and commercialize a live Web4 and AI-execution
infrastructure stack, not to fund a paper architecture.`
