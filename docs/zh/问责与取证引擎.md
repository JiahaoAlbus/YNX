# 问责 / 取证引擎

YNX 应该把当前 V2 的 trace layer 视为更大问责 / 取证引擎的基础层，而不是终局。

## 明确结论

结论很明确：你这次提的策略更好，但正确落地方式不是另起一个脱离现有系统的 V3，而是继续建立在 YNX 当前 V2 的 `lot lineage + pro-rata taint tracking` 之上。

我们采用下面这个分层：

- V2 基础证据层：`lot lineage + pro-rata taint tracking`
- 受保护的操作员 / 受害者访问层：挂在 Web4 policy/session 后面的 AI trace report 与 case flow
- V2 上层问责层：结构化 case、review log、escalation state、evidence chain
- 后续扩展层：更广的图遍历、更多标签提供器、更多可插拔检测器、多网络接入

这比“给每一个可替代币永久发一个人民币式唯一编号”更好，因为可替代资产在拆分、合并、混合之后，往往已经不再真实保留单一精确编号。继续硬说“这就是同一枚币”会制造伪精确。

所以正确原则是：

- 系统真的有精确来源关系时，就保留精确关系
- 已经发生合并 / 打散时，就保留比例来源关系
- 同时输出多种污染解释，而不是假装永远只有一种绝对解释

换句话说：这套策略更好，不是因为它更复杂，而是因为它对可替代资产更诚实。

## 这套策略在 YNX 里应该怎么解释

你给的这套策略方向是对的，但放进 YNX 里，正确理解不是“从零重写一套新系统”，而是继续把当前 V2 的 trace/accountability 栈往上补强。

按能力看，当前状态应该明确成这样：

- transaction flow graph：已经有，通过受保护的 `GET /trace/graph`
  支持上下游遍历、方向控制、hop limit、按金额/污染量筛选与 path
  summary
- lot lineage tracking：已经有，而且仍然是当前最核心的确定性证据层
- comparative taint models：已经有，当前输出 `poison`、`proRata`、
  `fifo`、`lifo`、`specificTrace`
- address clustering：已经有，case 创建时会输出启发式 cluster
- entity attribution：已经有，支持静态 label provider + 当前上下文推断标签
- suspicious pattern detection：已经有，当前覆盖 mixed exposure、root
  concentration、rapid multi-hop、amount-preserving hops、bridge-route
  exposure、pass-through、time-correlated routing、dormant-wallet
  reactivation
- risk scoring：已经有，会输出 explainable 的 severity、confidence、
  reasons
- evidence chain：已经有，而且 case 结论要求带证据链
- case report generator：已经有，能输出结构化 protected case 和 dossier
- API integration：已经有，挂在 AI Gateway 的 `ai.trace.report`、
  `ai.forensics.case.create`、`ai.forensics.case.review`、以及
  policy-scoped case read 上

当前还可以明确一点：

- protected case 不是“服务一重启就没了”的临时内存态
- 它会跟 AI gateway 其他状态一起持久化
- review status、escalation status、review logs 会在重启后保留
- `/health` 和 `/ready` 也会暴露 case 聚合状态，方便运维核对线上/本地是否一致

真正还要继续扩展的主要是：

- 更广的多账本 / 多网络数据接入
- 更多 attribution provider / label dataset
- 更多 detector family 和 operator workflow
- 如果以后需要更强独立存储，再补更正式的 persistence / migration 层

## 为什么不是“每个币一个永久编号”

像人民币那样的单张编号，适用于天然离散、不会链上自动合并成同质余额的对象。

但在链上，尤其是账户模型里：

- 多笔同资产会并入同一余额
- 一笔支出常常来自多个来源一起出资
- 一笔入账之后又可能被继续拆分

这时如果还坚持“每个最小单位永远一币一码”，会有三个问题：

1. 合并后必须伪造一个不存在的唯一身份
2. 拆分后必须伪造“母币”和“子币”的绝对继承关系
3. 用户和审计方会误以为系统拿到了现实中并不存在的精确事实

而 `lot lineage + pro-rata taint` 的表达是：

- 入账时创建或扩展一个来源 lot
- 花费时从父 lot 派生子 lot
- 拆分保留父子关系
- 合并保留多个父来源及其占比
- 风险与污染按模型分别解释

这样即使币被“打散”，系统也不会丢掉问责能力，只是从“唯一编号叙事”切换为“来源向量 + 占比 + 证据链叙事”。

## 如果还想更强一点，应该怎么加

如果你希望比现在更强，但又不想走错架构，推荐这样增强：

- 给每个初始 lot 增加不可变的 `issuance_id` 或 `deposit_batch_id`
- 对桥入、铸造、托管入金、运营商标记事件保留独立 `origin_ref`
- 在 case report 里同时输出：
  - 精确父 lot 关系
  - 比例污染结果
  - 证据交易路径
  - 人工复核意见

这会带来更好的效果：

- 初始来源更稳定，可长期引用
- 桥接 / 入金 / 申诉场景里更容易对账
- 不会伪造“每个最终余额分子都有唯一终身编号”

所以更好的增强不是“每个币永久编号”，而是“每个来源 lot / 入金批次有稳定编号 + 后续沿 lineage 传播”。

目前 YNX 已经开始按这个方向落地：

- traced lot 可携带 `issuance_id`
- traced lot 可携带 `deposit_batch_id`
- child lot 会继承这些来源锚点
- case report / evidence chain / dossier 也会带出这些锚点

所以如果你问“这份策略要不要采用”，更准确的正式答案应该是：

- 要采用
- 但要按 YNX 当前架构改写后采用
- 不是脱离现有系统重新造一个 V3
- 也不是改成“每个可替代币永久一币一码”
- 而是继续把当前 V2 证据层升级成更强的 protected accountability
  surface

## V2 还是 V3

明确建议：

- 这件事应该继续放在 V2 里做强
- 不应该因为要做问责 / 取证就单独拆成一个概念性 V3

原因是：

- 当前 lot lineage 已经是现成的确定性证据底座
- AI case、review、escalation 已经挂在现有 Web4 / AI 网关语义里
- 如果现在强行独立成 V3，反而会把证据层、操作层、对外口径拆散

只有在下面情况都出现时，才值得把它单独升级成更大的版本层：

- 需要统一接入多条外部链和多种账本模型
- 需要跨网络标准化证据交换
- 需要更独立的取证存储、索引、审计证明体系

也就是说：

- 当前建议：`V2 trace + V2 accountability`
- 未来可能：`V3 multi-network forensic fabric`

## 目前已经适合做什么

YNX 当前最适合的方向是：

- 指定资产的欺诈追踪
- 来源重建
- 风险解释
- 受保护的 case 处理
- 面向受害者 / 操作员 / 合规的证据报告

而不应夸大成：

- 全资产、全网络、全场景的终局取证平台
- 每个币最小分子永久一币一码
- 自动冻结 / 自动没收系统

## 安全边界

取证层只负责观察、解释、留证、升级。

它不授予：

- 私自转走资产的权限
- 私自冻结资产的权限
- 私自封禁账户的权限
- 绕过合规审批的执行权

推荐流程：

1. 用户或操作员发起受保护 trace / case
2. AI 生成可读摘要与证据链
3. 人工复核
4. 如需冻结、上报、请求交易所协查，再走独立审批路径

## 当前 YNX 的正确技术口径

目前对外最准确的说法应该是：

- YNX V2 已经有 lot lineage + pro-rata taint tracking
- 它已经能为更高层的问责 / 取证 case 提供证据底座
- YNX 适合继续在 V2 上补强 clustering、labeling、pattern detection、risk scoring、case dossier
- 不应该把系统描述成“每个可替代币最终都带永久唯一编号”

## 后续建议

1. 继续保留 `lot lineage + pro-rata taint` 作为默认真相层
2. 给初始来源 lot 增加更稳定的 `issuance_id` / `deposit_batch_id`
3. 持续扩展标签提供器、图检测器、case dossier
4. 未来真的要做多链统一取证时，再把它升格成更大的版本层
